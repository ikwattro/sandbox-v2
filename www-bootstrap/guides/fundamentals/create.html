Cypher 101


  • Ascii Art + Query Language = Cypher

  • Made for Graphs

  • Declarative, Readable

  • Focus on Intent, the HOW

  • Describe the patterns you want to discover

  • Readable, Concise

  • Powerful Capabilities

Nodes


() or (n)

  • Surrounded with Parentheses

  • Use an Alias to refer to our node later

(n:Label)

  • Specify a Label - grouping nodes by roles or types

  • Think of Labels like "Tables++"

(n:Label {prop: 'value'})

  • Nodes can have Properties

  • Properties are stored as key-value pairs

  • Strings, Numbers, Booleans, and arrays thereof

Relationships


()-->() or ()-[:TYPE]->()

  • Encased with Hypens & Square Brackets

  • Like labels, a Relationship Type starts with a colon, i.e. :

  • < or > specify the direction of the relationship

()-[:KNOWS {since: 2010}]->()

  • can have properties too!

Constraints


Before we get started…​

  • I want to make sure that every :Person is special in our graph!!

Constraints


  • Assert uniqueness

  • For a given property

  • On a particular label

CREATE CONSTRAINT ON (p:Person) ASSERT p.name IS UNIQUE;

  • Everyones name property will be have to be unique!

Your turn - CREATE yourself!


  • CREATE a person node within the graph to represent yourself!

  • Label of :Person

  • Property name should be 'Your Name'

Welcome to the graph!


  • CREATE statement:

CREATE (p:Person {name: 'Your Name'})
RETURN p;

MATCH - Find yourself


  • Starting point for our query

  • MATCH patterns in the graph

MATCH (p:Person {name: 'Your Name'})
RETURN p;
MATCH (p:Person)
WHERE p.name='Your Name'
RETURN p;

SET - Add and Update Properties


Find yourself and SET your city as property

MATCH (p:Person {name: 'Your Name'})
SET p.city = 'Your City'
RETURN p;

Teaser of things to come


Listing people and cities

MATCH (p:Person)
RETURN p.name, p.city;

Aggregation by City

MATCH (p:Person)
RETURN p.city, collect(p.name) as people;

Top Cities

MATCH (p:Person)
RETURN p.city, count(*) as count, collect(p.name) as people
ORDER BY count DESC LIMIT 5;

MERGE - Get-Or-Create


Let’s create us twice, what happens?

CREATE (:Person {name:'Your Name'});

MERGE for the rescue

  • MATCH to check the whole pattern exists

  • If not, Cypher will CREATE it

ON CREATE / MATCH SET - Conditionals


We can use MERGE with SET conditions too

ON CREATE SET

MERGE (p:Person {name: 'Johan'})
ON CREATE SET p.city = 'Malmö';

ON MATCH SET

MERGE (p:Person {name: 'Johan'})
ON MATCH SET p.boss = true ;

Creating Relationships


  • Ask the person to your right for their name

  • Find you both in the graph

MATCH (p:Person {name: 'Your Name' })
MATCH (o:Person {name: 'Other Name'})
RETURN p,o;

In a single MATCH clause.

MATCH (p:Person {name: 'Your Name' }) ,
      (o:Person {name: 'Other Name'})
RETURN p,o;

Creating Relationships


MATCH  (p:Person {name: 'Your Name' })
MATCH  (o:Person {name: 'Other Name'})
CREATE (p)-[:KNOWS]->(o);

  • MATCH a person, name equal 'Your Name'

  • MATCH a person, name equal 'Other Name'

  • CREATE a relationship of type :KNOWS

MERGE-ing Relationships


MATCH (p:Person {name: 'Your Name'})
MATCH (p:Person {name: 'Other Name'})
MERGE (p)-[:KNOWS]->(o)
RETURN p,o;

  • Will create relationship uniquely

  • per type, direction and properties

MERGE-ing Patterns


Partially bound patterns are created in context.

MATCH (p:Person {name: 'Your Name' })
MERGE (p)-[:WROTE]->(a:Article {title: 'My first Cypher'})
RETURN p,a;

  • Cypher will MATCH the whole pattern

  • If not found CREATE it

DELETE - Clean Up


What will happen?

MATCH (p:Person {name: 'Your Name'}) DELETE p;

  • No Dangling Pointers!

  • A Relationship must always have a start and end-node

  • Delete node and relationships

DETACH DELETE


MATCH (p:Person {name: 'Your Name'}) DETACH DELETE p;

  • DETACH node by deleting all relationships

  • DELETE the node
